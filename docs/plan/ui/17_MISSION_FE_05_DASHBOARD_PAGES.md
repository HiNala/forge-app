# MISSION FE-05 — Dashboard, Page Detail, Submissions & Automations UI

**Goal:** Build the managerial surface — the screens Lucy uses after the page is made. Dashboard shows her all her pages at a glance with unread counts and status. Page Detail is where she reads submissions, wires up automations, checks performance. The Submissions table has to make reading and replying to 50 submissions as painless as 5. After this mission, Forge is no longer a content creator's toy — it's a small-business intake system.

**Branch:** `mission-fe-05-dashboard-pages`
**Prerequisites:** FE-04 complete. Studio works. Backend Missions 04 and 05 complete (submissions flow, automations engine, calendar OAuth).
**Estimated scope:** Medium-large. Several complex table and form surfaces.

---

## The Mixture of Experts Lens for This Mission

- **Nielsen** — *"Can users accomplish tasks quickly? Is system status always visible?"* Unread counts must be correct and instant. Filter state must match the URL.
- **Tesler** — *"Can actions be reversed? Can users explore safely?"* Bulk archive must be undoable. Replying must show a clear "this has been sent" state.
- **Engelbart** — *"What happens after the first action?"* After a submission arrives, Lucy's next action is obvious — reply, or mark read, or set up an automation.
- **Fadell** — *"What happens before and after use? Where are hidden frictions?"* The moment between "submission arrives" and "Lucy replies" should feel like one motion, not five.
- **Norman** — *"Are possible actions obvious from appearance alone?"* A row-hover state exposes actions. An unread dot is unmistakable.
- **Ive** — *"Does this respect the user's attention?"* No sidebar inside Page Detail. Horizontal tabs only.

## How To Run This Mission

Read User Case Flows 4, 5, 6, 7, 8, and 10 from the user case reports. These define every interaction in this mission.

The defining interaction here is the **inline expand-in-place** pattern on the submissions table. No row ever navigates to a separate page. Click the row, it expands, all data is visible, all actions are one click away, collapse restores the list. If a submission has 8+ fields, show the first 6 with a "see all" button.

Commit on each screen completion: Dashboard with grid and filters, Page Detail shell with tabs, Overview tab, Submissions table with expand, reply flow, bulk actions, Automations UI, calendar connection flow.

27. Only one row expanded at a time. Opening another collapses the first.
28. Bulk selection: when any checkbox is checked, a sticky action bar appears at the bottom: "{n} selected" + Mark as read · Archive · Delete · Cancel.
29. Bulk delete confirms via Dialog with typed confirmation for 10+ items.
30. Empty state: "No submissions yet" with a "Share your page" helper showing the live URL.
31. Loading state: 8 skeleton rows.
32. Real-time updates: subscribe to a lightweight SSE channel (`/api/v1/pages/{id}/submissions/stream`) OR poll every 30s. New rows appear at the top with a gentle `fadeUp` + highlight pulse. Unread count in the tab and the sidebar updates instantly.

### Phase 5 — Reply Flow

33. Reply button opens a Dialog with: subject input (pre-filled "Re: your submission"), body textarea (pre-filled with a contextual AI-drafted reply), "Regenerate draft" button, Cancel + Send.
34. "Regenerate draft" calls `POST /api/v1/submissions/{id}/draft-reply` which uses the fast LLM to produce a suggested reply based on the submission. Replaces the textarea content.
35. Send calls `POST /api/v1/submissions/{id}/reply`. On success: toast "Reply sent to {email}", submission row updates to "replied" status, Dialog closes.
36. Below the compose area: a subtle preview of how the email will look with the brand kit applied (logo, primary color accent).
37. If the reply fails (Resend down), toast shows error with a "Try again" action.

### Phase 6 — CSV Export

38. "Export CSV" button calls `GET /api/v1/pages/{id}/submissions/export?format=csv` with the current filter + search state applied. Streams download.
39. While download pending: button shows a small spinner.
40. On completion: toast with a filename confirmation and a "Download again" retry action.

### Phase 7 — Automations Tab

41. `(app)/pages/[pageId]/automations/page.tsx`. Warm, clean layout — not a form in the traditional sense. Feels like a settings panel.
42. Three sections, each a Card:
    - **Notify on submission** — big toggle. When on: a chip-list editor of email addresses (enter to add, click × to remove, max 5). Below: a small preview of the notification email with the brand kit applied.
    - **Send confirmation to submitter** — big toggle. When on: editable subject input, editable body textarea, live email preview pane to the right. Changes auto-save (debounced 500ms). Show a small "Saved" checkmark 2 seconds after change.
    - **Sync to calendar** — big toggle. When off: shows "Connect Google Calendar" and "Connect Apple Calendar (coming soon)" buttons. When connected: shows connected calendar name + disconnect link, plus sub-options (auto-create event toggle, event duration selector 15/30/45/60/90/120/custom minutes, "Send invite to submitter" toggle, "If form has a date field, use that" toggle).
43. Below all three sections: a small "Recent runs" section listing the last 10 AutomationRun rows with status dots and timestamps. Click a failed run to see the error message and a "Retry" button.

### Phase 8 — Calendar OAuth Connection Flow

44. "Connect Google Calendar" button opens a new window to `/api/v1/calendar/connect/google`. The backend redirects to Google's consent screen.
45. After consent, Google redirects back to `/api/v1/calendar/callback/google`, which closes the popup and posts a message to the opener window: `{type: 'calendar:connected', provider: 'google'}`.
46. The Automations page listens for the message and re-fetches the calendar connections list. UI updates without a refresh.
47. If the OAuth fails or is denied: popup closes, parent shows a toast "Google Calendar connection failed. You can try again anytime."
48. If the user closes the popup mid-flow: detect via a setInterval check on `popup.closed`; show toast "Connection cancelled."

### Phase 9 — Page Actions Menu

49. The ⋮ overflow menu in the Page Detail header:
    - **Duplicate** → `POST /api/v1/pages/{id}/duplicate` → navigates to Studio with the new draft.
    - **Archive** → confirms in a Dialog ("This page will stop accepting submissions. You can restore it anytime."), then sets status to archived.
    - **Delete** → confirms with a typed-confirmation for live pages (type "delete" to confirm). Deletes submissions after a 30-day grace period per PRD.
    - **View versions** → opens a Sheet listing PageVersions with a "Revert to this version" action on each.
    - **Manage custom domain** → opens a Dialog for custom-domain setup (covered in Mission 04 backend).
